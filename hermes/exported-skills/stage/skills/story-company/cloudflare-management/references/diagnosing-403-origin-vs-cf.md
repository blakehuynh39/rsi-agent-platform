# Diagnosing Origin 403 vs. Cloudflare WAF Block

When an internal endpoint returns HTTP 403, the response **body** is the key differentiator.

## Empty 403 ŌåÆ origin-level rejection

An **empty body** 403 means the request reached the origin and the origin's auth layer rejected it. The response is generated by `StatusCode::FORBIDDEN.into_response()` in Axum with no body.

In depin-backend NDV endpoints, this specifically means **`numo_validation.enabled = false`**:

```rust
// extractors_numo_validation.rs:85-87
if !cfg.enabled {
    return Err(AuthError::Disabled);  // ŌåÆ 403, empty body
}
```

Other depin-backend auth rejections return different status codes:
| Error | HTTP | Body |
|-------|------|------|
| `AuthError::Disabled` | 403 | **empty** |
| `AuthError::TokenMissing` | 401 | empty |
| `AuthError::TokenInvalid` | 401 | empty |
| `AuthError::Internal` | 500 | empty |

**Root cause checklist for empty 403 on NDV endpoints:**

1. `NUMO_VALIDATION_ENABLED` env var ŌĆö overrides the TOML config at runtime (`config.rs:859-860`). If set to `false` (or unset with a loader quirk), the feature is disabled.
2. `numo_validation.enabled` in `production.toml` ŌĆö but this defaults to `true` via `base.toml` and `#[serde(default = "default_numo_validation_enabled")]`.
3. Service restart ŌĆö if env vars were changed but the service wasn't redeployed/restarted.

**Environment variable override (config.rs:859-860):**

```rust
if let Some(value) = env_parse("NUMO_VALIDATION_ENABLED")? {
    self.numo_validation_mut().enabled = Some(value);
}
```

This env var takes precedence over all TOML config files. Both staging and production deployments of `depin-api` must have `NUMO_VALIDATION_ENABLED=true` for NDV endpoints to work.

Also required:
- `NUMO_VALIDATION_SERVICE_TOKEN_SHA256` ŌĆö 64-char hex SHA-256 of the NDV Bearer token
- `NUMO_VALIDATION_SERVICE_TOKEN_SHA256_NEXT` (optional) ŌĆö for rotation overlap

## HTML 403 ŌåÆ Cloudflare WAF block

If the response body contains HTML with `"Sorry, you have been blocked"` and a `#cf-footer-ip` span, Cloudflare's WAF blocked the request **before** it reached origin. Use the standard WAF diagnosis flow in the parent skill.

The CF Ray ID is still present in headers in both cases ŌĆö it doesn't distinguish origin rejection from CF block. The body does.

## 403 with other body content

Some Cloudflare managed rules (like OWASP) may return a terse 403 with a different body format. Check the `server: cloudflare` response header ŌĆö if present, CF handled the response. If absent (or shows the origin server), the origin returned it.

## Operational access constraints

**RSI cannot directly modify production K8s deployments.** The production `depin-backend` deployment is NOT in the `story` or `rsi-platform` Kubernetes namespaces (only staging `use1-stage-depin-backend` is visible). The production ALB (`use1-prod-depin-backend-520197523.us-east-1.elb.amazonaws.com`, from `cloudflare/records.ts:35`) exists but its backend targets are in a separate cluster/namespace outside RSI's read scope.

**What RSI CAN do for production fixes:**
- Create PRs in `piplabs/depin-backend` to fix config files (e.g., `production.toml`)
- Create PRs in `piplabs/cloudflare` to adjust WAF/DNS rules (these apply to both staging and production via `${API_HOSTS}`)
- Diagnose issues from code analysis and CF Ray ID evidence
- After PR merge, the production deployment must be updated/restarted by someone with production K8s access

**PR #449** is the reference pattern for production config fixes: added `enabled = true` explicitly to `production.toml`'s `[numo_validation]` section (the file had an empty section header with no values).
