# Diagnosing Origin 403 vs. Cloudflare WAF Block

When an internal endpoint returns HTTP 403, the response **body** is the key differentiator.

## Empty 403 → origin-level rejection

An **empty body** 403 means the request reached the origin and the origin's auth layer rejected it. The response is generated by `StatusCode::FORBIDDEN.into_response()` in Axum with no body.

In depin-backend NDV endpoints, this specifically means **`numo_validation.enabled = false`**:

```rust
// extractors_numo_validation.rs:85-87
if !cfg.enabled {
    return Err(AuthError::Disabled);  // → 403, empty body
}
```

Other depin-backend auth rejections return different status codes:
| Error | HTTP | Body |
|-------|------|------|
| `AuthError::Disabled` | 403 | **empty** |
| `AuthError::TokenMissing` | 401 | empty |
| `AuthError::TokenInvalid` | 401 | empty |
| `AuthError::Internal` | 500 | empty |

**Root cause checklist for empty 403 on NDV endpoints:**

1. `NUMO_VALIDATION_ENABLED` env var — overrides the TOML config at runtime (`config.rs:859-860`). If set to `false` (or unset with a loader quirk), the feature is disabled.
2. `numo_validation.enabled` in `production.toml` — but this defaults to `true` via `base.toml` and `#[serde(default = "default_numo_validation_enabled")]`.
3. Service restart — if env vars were changed but the service wasn't redeployed/restarted.

**Environment variable override (config.rs:859-860):**

```rust
if let Some(value) = env_parse("NUMO_VALIDATION_ENABLED")? {
    self.numo_validation_mut().enabled = Some(value);
}
```

This env var takes precedence over all TOML config files. Both staging and production deployments of `depin-api` must have `NUMO_VALIDATION_ENABLED=true` for NDV endpoints to work.

Also required:
- `NUMO_VALIDATION_SERVICE_TOKEN_SHA256` — 64-char hex SHA-256 of the NDV Bearer token
- `NUMO_VALIDATION_SERVICE_TOKEN_SHA256_NEXT` (optional) — for rotation overlap

## HTML 403 → Cloudflare WAF block

If the response body contains HTML with `"Sorry, you have been blocked"` and a `#cf-footer-ip` span, Cloudflare's WAF blocked the request **before** it reached origin. Use the standard WAF diagnosis flow in the parent skill.

The CF Ray ID is still present in headers in both cases — it doesn't distinguish origin rejection from CF block. The body does.

## 403 with other body content

Some Cloudflare managed rules (like OWASP) may return a terse 403 with a different body format. Check the `server: cloudflare` response header — if present, CF handled the response. If absent (or shows the origin server), the origin returned it.
